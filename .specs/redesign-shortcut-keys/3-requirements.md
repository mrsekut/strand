# Requirements: Shortcut Key Redesign

## User Stories

1. ユーザーとして、一覧画面と詳細画面で独立したキーバインドを持ちたい。画面ごとに適切な操作だけが有効になるようにしたい。
2. ユーザーとして、矢印キーでissue間を移動したい。`j`/`k` は廃止し、詳細画面でのテキスト閲覧中の誤操作を防ぎたい。
3. ユーザーとして、AI操作を `a` 接頭辞でまとめたい。`ae` でenrich、`ai` でimplementを実行したい。
4. ユーザーとして、詳細画面から `Esc` で一覧に戻りたい。`Enter` は「開く」専用にし、「戻る」には使わない。

## Acceptance Criteria

### 画面別キー分離 (strand-gc3)

- [ ] 一覧画面のキー入力は一覧画面専用のハンドラで処理される
- [ ] 詳細画面のキー入力は詳細画面専用のハンドラで処理される
- [ ] 共通キー（`q`, `Enter`）は両画面で動作する

### ナビゲーション変更 (strand-cco)

- [ ] `j`/`k` キーによるissue移動が削除されている
- [ ] `↑`/`↓` 矢印キーで一覧画面のissue移動ができる
- [ ] 詳細画面では `↑`/`↓` で前後のissueに移動できる

### Enter/Esc分離 (strand-nuq)

- [ ] 一覧画面で `Enter` を押すと詳細画面に遷移する
- [ ] 詳細画面で `Esc` を押すと一覧画面に戻る
- [ ] 詳細画面で `Enter` は「戻る」として機能しない
- [ ] 一覧画面で `Esc` は何もしない（または終了確認なし）

### AI操作の接頭キー (strand-0ln)

- [ ] `a` キー押下で「AI操作待ち」状態になる
- [ ] `a` → `e` で enrich が実行される（一覧画面のみ）
- [ ] `a` → `i` で implement が実行される（一覧画面のみ）
- [ ] `a` の後に無効なキーを押した場合、待ち状態がキャンセルされる
- [ ] 待ち状態であることがUIに表示される（ステータスバー等）

### UIヘルプ表示

- [ ] 一覧画面のヘルプが新しいキーバインドを反映している
- [ ] 詳細画面のヘルプが新しいキーバインドを反映している

## キーバインド設計

### 一覧画面

| キー | アクション |
|------|-----------|
| `q` | 終了 |
| `↑`/`↓` | issue移動 |
| `Enter` | 詳細画面へ |
| `c` | issue IDコピー |
| `ae` | AI enrich |
| `ai` | AI implement |

### 詳細画面

| キー | アクション |
|------|-----------|
| `q` | 終了 |
| `Esc` | 一覧画面へ戻る |
| `↑`/`↓` | 前後のissueに移動 |
| `c` | issue IDコピー |
| `e` | title+description編集 |
| `m` | impl merge (impl done時のみ) |
| `d` | impl discard (impl done時のみ) |

## Scope

### In Scope

- キー入力ハンドラの画面別分離
- `j`/`k` の廃止と矢印キーへの統一
- `a` 接頭辞による2キーシーケンス実装
- UIヘルプテキストの更新

### Out of Scope

- キーバインドのカスタマイズ機能
- 新しいアクションの追加
- UI レイアウトの変更
